# Phase 02.1: Editor Edit Mode + Diff View - Research

**Researched:** 2026-03-25
**Domain:** TipTap 3 editor migration, edit/save/search, ProseMirror decorations for diff view
**Confidence:** HIGH

## Summary

This phase replaces the react-markdown read-only viewer with a TipTap editor that supports both read-only and edit modes, adds save/search/undo-redo functionality, and implements an inline diff overlay for rejection feedback. The most significant discovery is that TipTap has progressed to v3 (currently 3.20.5) since the project spec was written. TipTap 3 includes breaking changes from v2: package consolidation (TaskList/TaskItem into @tiptap/extension-list, Placeholder/UndoRedo into @tiptap/extensions), an official @tiptap/markdown extension replacing the community tiptap-markdown package, and API changes (History renamed to UndoRedo, shouldRerenderOnTransaction defaults to false in React). Since no TipTap v2 code exists in the codebase (Phase 2 used react-markdown), we install TipTap 3 directly with no migration cost.

The diff overlay uses ProseMirror's Decoration.inline() API to apply green/red CSS classes to changed text ranges. The `diff` library's diffLines() computes line-level changes between snapshot and updated content. For search, since no official TipTap search extension exists, we wrap the `prosemirror-search` package (v1.1.0) as a TipTap extension, which provides query management, match highlighting via DecorationSet, and find-next/find-prev commands.

**Primary recommendation:** Install TipTap 3.20.5 with @tiptap/markdown (official, free). Use prosemirror-search for find-in-document. Build diff overlay as a custom ProseMirror plugin using Decoration.inline(). Extend EditorTab type with mode/isDirty fields and add POST /api/files/save endpoint.

### Pattern 1: TipTap Editor with Mode Toggle
**What:** A single TipTap editor instance that toggles between read-only and editable states via `editor.setEditable()`.
**When to use:** When the same document needs viewing and editing.
**Example:**
```typescript
// Source: TipTap 3 docs — editor.setEditable()
import { useEditor, EditorContent } from '@tiptap/react';
import StarterKit from '@tiptap/starter-kit';
import { Markdown } from '@tiptap/markdown';
import { TaskList, TaskItem } from '@tiptap/extension-task-list';
import { Placeholder } from '@tiptap/extension-placeholder';
import CodeBlockLowlight from '@tiptap/extension-code-block-lowlight';

const editor = useEditor({
  extensions: [
    StarterKit.configure({
      codeBlock: false, // replaced by CodeBlockLowlight
    }),
    Markdown,
    TaskList,
    TaskItem.configure({ nested: true }),
    Placeholder.configure({ placeholder: 'No content' }),
    CodeBlockLowlight.configure({ lowlight }),
  ],
  content: markdownString,
  contentType: 'markdown',
  editable: false, // read-only by default
  shouldRerenderOnTransaction: false, // TipTap 3 React performance default
});

// Toggle mode:
editor.setEditable(true);  // enter edit mode
editor.setEditable(false); // return to read-only

// Get markdown for saving:
const markdown = editor.getMarkdown();

// Set content from markdown:
editor.commands.setContent(newMarkdown, { contentType: 'markdown' });
```

### Pattern 2: ProseMirror Decoration-Based Diff Overlay
**What:** A TipTap extension that applies Decoration.inline() for added/removed lines.
**When to use:** After file change events, to show what changed visually.
**Example:**
```typescript
// Source: ProseMirror docs — Decoration.inline()
import { Extension } from '@tiptap/core';
import { Plugin, PluginKey } from '@tiptap/pm/state';
import { Decoration, DecorationSet } from '@tiptap/pm/view';

const diffPluginKey = new PluginKey('diffOverlay');

const DiffOverlay = Extension.create({
  name: 'diffOverlay',
  addProseMirrorPlugins() {
    return [
      new Plugin({
        key: diffPluginKey,
        state: {
          init: () => DecorationSet.empty,
          apply: (tr, oldSet) => {
            const meta = tr.getMeta(diffPluginKey);
            if (meta?.decorations) return meta.decorations;
            if (meta?.clear) return DecorationSet.empty;
            return oldSet.map(tr.mapping, tr.doc);
          },
        },
        props: {
          decorations(state) {
            return diffPluginKey.getState(state);
          },
        },
      }),
    ];
  },
});
```

### Pattern 3: Wrapping prosemirror-search as TipTap Extension
**What:** A TipTap extension that wraps the prosemirror-search plugin for find-in-document.
**When to use:** For Cmd+F search with match highlighting.
**Example:**
```typescript
// Source: prosemirror-search docs + TipTap addProseMirrorPlugins
import { Extension } from '@tiptap/core';
import { search, SearchQuery, setSearchState, findNext, findPrev } from 'prosemirror-search';

const SearchExtension = Extension.create({
  name: 'search',
  addProseMirrorPlugins() {
    return [search()];
  },
  addCommands() {
    return {
      setSearchQuery: (query: string) => ({ tr, dispatch }) => {
        if (dispatch) {
          const searchQuery = new SearchQuery({ search: query, caseSensitive: false });
          setSearchState(tr, searchQuery);
          dispatch(tr);
        }
        return true;
      },
      clearSearch: () => ({ tr, dispatch }) => {
        if (dispatch) {
          setSearchState(tr, new SearchQuery({ search: '' }));
          dispatch(tr);
        }
        return true;
      },
    };
  },
  addKeyboardShortcuts() {
    return {
      'Mod-f': () => {
        // Emit event or set state to show search bar
        return true; // prevent default browser search
      },
    };
  },
});
```

### Pattern 4: File Watcher Suppression for Self-Saves
**What:** Prevent the file watcher from broadcasting a change event when the save originated from the editor itself.
**When to use:** POST /api/files/save endpoint.
**Example:**
```typescript
// Backend approach: short debounce window after save
// In file-watcher.ts, maintain a set of recently-saved paths
const recentlySaved = new Set<string>();

// In POST /api/files/save handler:
recentlySaved.add(relPath);
await writeFile(resolvedPath, content, 'utf-8');
setTimeout(() => recentlySaved.delete(relPath), 500);

// In file watcher event handler:
watcher.on('change', async (filePath) => {
  const relPath = relative(projectRoot, filePath);
  if (recentlySaved.has(relPath)) return; // suppress self-save event
  // ... normal broadcast
});
```

**Alternative (client-side approach):** Set a flag in Zustand store before save, ignore next filesystem:changed for that path, clear flag after event received. Both approaches are valid; backend approach is simpler and more reliable.

### Pattern 5: Snapshot Capture for Diff
**What:** Before sending a chat message, snapshot all open tab contents for later comparison.
**When to use:** On chat submit (handleSend in ChatView).
**Example:**
```typescript
// In Zustand store:
snapshotTabs: Map<string, string>; // filePath -> content at snapshot time
captureSnapshot: () => void;
getSnapshot: (filePath: string) => string | undefined;
clearSnapshot: () => void;

// In ChatView handleSend:
const handleSend = useCallback(() => {
  // Capture snapshot BEFORE sending message
  useAppStore.getState().captureSnapshot();
  sendMessage({ text });
}, []);

// In useWebSocket filesystem:changed handler:
if (payload.type === 'changed') {
  const snapshot = store.getSnapshot(payload.path);
  if (snapshot && snapshot !== payload.content) {
    // Compute diff and activate overlay
    store.setDiffData(payload.path, computeDiff(snapshot, payload.content));
  }
  store.clearSnapshot(); // one-shot
}
```

### Anti-Patterns to Avoid
- **Re-creating editor on content update:** Do NOT destroy and recreate the TipTap editor when content changes. Use `editor.commands.setContent()` instead. Creating a new editor loses undo history, cursor position, and selection.
- **Using editor.getHTML() for save:** This produces HTML, not markdown. Always use `editor.getMarkdown()` when saving to .md files.
- **Relying on useEditor editable option for runtime toggle:** In TipTap 3, changes to the `editable` prop passed to `useEditor` may not reactively update the editor. Use `editor.setEditable()` programmatically instead.
- **Ignoring shouldRerenderOnTransaction:** In TipTap 3, this defaults to false. If toolbar state depends on editor state (e.g., bold active), use `useEditorState` selector hook instead of relying on component re-renders.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Markdown editing | Custom contentEditable + markdown parser | TipTap 3 + @tiptap/markdown | ProseMirror handles selection, cursor, input events, composition, IME. Thousands of edge cases. |
| Search with highlighting | Custom text search + DOM manipulation | prosemirror-search | Handles decorations across node boundaries, active match tracking, wrap-around, regex support |
| Text diffing | Custom line-by-line comparison | diff library (diffLines) | Handles edge cases: trailing newlines, empty lines, moved blocks, optimal diff algorithm |
| Undo/redo | Custom history stack | StarterKit includes UndoRedo extension | ProseMirror's undo tracks operations, not snapshots. Handles selection restoration, composition events. |
| Scroll preservation | Manual scrollTop save/restore | ProseMirror's scroll-into-view mechanism | ProseMirror maps positions through document changes, scrollIntoView() is transaction-aware |

**Key insight:** ProseMirror and TipTap handle an enormous amount of browser-specific complexity (selection APIs, IME input, clipboard, drag/drop). Every hand-rolled solution in this domain will have subtle bugs in edge cases that ProseMirror has already solved.

## Common Pitfalls

### Pitfall 1: TipTap 3 vs TipTap 2 Package Names
**What goes wrong:** Installing @tiptap/extension-task-list@2.x (old separate packages) instead of 3.x, or importing from wrong consolidated packages.
**Why it happens:** CLAUDE.md references TipTap ^2.11. The npm registry has both 2.x and 3.x versions.
**How to avoid:** Always install @tiptap/*@3.20.5 explicitly. In TipTap 3, TaskList and TaskItem are still available as separate packages (@tiptap/extension-task-list, @tiptap/extension-task-item) even though they're also re-exported from @tiptap/extension-list. Use the separate packages for simpler imports. Placeholder is also available from both @tiptap/extension-placeholder and @tiptap/extensions.
**Warning signs:** TypeScript version mismatch errors, "Cannot find module" for extension imports.

### Pitfall 2: Content Type Not Specified When Loading Markdown
**What goes wrong:** Editor renders raw markdown as text instead of parsed rich content.
**Why it happens:** Without `contentType: 'markdown'`, TipTap treats the string as HTML.
**How to avoid:** Always pass `contentType: 'markdown'` when using setContent() with markdown strings: `editor.commands.setContent(markdown, { contentType: 'markdown' })`. For initial content, pass it in useEditor options with `contentType: 'markdown'`.
**Warning signs:** Seeing literal # and ** characters in the rendered document.

### Pitfall 3: Self-Save Reload Loop
**What goes wrong:** User saves file -> file watcher detects change -> WebSocket pushes update -> editor reloads content -> triggers onChange -> marks as dirty.
**Why it happens:** chokidar fires a change event for any file write, including the editor's own saves via POST /api/files/save.
**How to avoid:** Implement save suppression (see Pattern 4). Use a backend-side recently-saved path set with short TTL, or client-side ignore-next-change flag.
**Warning signs:** Editor content flickering after save, or dirty indicator reappearing immediately after saving.

### Pitfall 4: Scroll Position Lost on setContent
**What goes wrong:** When live-reload updates editor content via setContent(), the scroll position jumps to the top.
**Why it happens:** ProseMirror's setContent replaces the entire document, invalidating all scroll state.
**How to avoid:** Before calling setContent, capture the current scroll position of the editor's DOM element. After setContent, restore the scrollTop. Use `editor.view.dom.parentElement.scrollTop` (the overflow container) rather than `editor.view.dom.scrollTop`.
**Warning signs:** Document jumping to top on every file change event.

### Pitfall 5: Diff Decorations Break After Content Update
**What goes wrong:** Decoration positions become invalid after the document changes, causing incorrect highlighting or crashes.
**Why it happens:** ProseMirror Decorations are position-based. When the document changes, positions shift.
**How to avoid:** Use the plugin's `apply()` method to map decorations through transactions: `oldSet.map(tr.mapping, tr.doc)`. When setting new content, clear decorations first, then recompute. Never hold stale DecorationSet references.
**Warning signs:** Decorations appearing on wrong text, or console errors about invalid positions.

### Pitfall 6: Conflict Between Edit Mode and Live-Reload
**What goes wrong:** User is editing a document when a filesystem:changed event arrives, silently overwriting their changes.
**Why it happens:** The current useWebSocket handler calls updateTabContent unconditionally on filesystem:changed.
**How to avoid:** Check if the tab is in edit mode and isDirty before updating. If dirty, show a conflict warning instead of silently overwriting. If read-only (or not dirty), update normally.
**Warning signs:** User loses typed content without warning.

### Pitfall 7: Missing EditorTab State Fields
**What goes wrong:** Store references setTabMode or setTabDirty but these actions don't exist.
**Why it happens:** The CONTEXT.md states "Zustand store already has setTabMode and setTabDirty actions from Phase 2" -- but inspection of the actual store code reveals these were never implemented. Phase 2 did not implement edit mode.
**How to avoid:** Add `mode: 'read' | 'edit'` and `isDirty: boolean` fields to the EditorTab type in shared/src/types/editor.ts. Add `setTabMode` and `setTabDirty` actions to the Zustand store.
**Warning signs:** TypeScript errors when accessing tab.mode or tab.isDirty.

### TipTap Editor Initialization with All Extensions
```typescript
// Source: TipTap 3 docs + verified package APIs
import { useEditor, EditorContent } from '@tiptap/react';
import StarterKit from '@tiptap/starter-kit';
import { Markdown } from '@tiptap/markdown';
import TaskList from '@tiptap/extension-task-list';
import TaskItem from '@tiptap/extension-task-item';
import Placeholder from '@tiptap/extension-placeholder';
import CodeBlockLowlight from '@tiptap/extension-code-block-lowlight';
import { common, createLowlight } from 'lowlight';

const lowlight = createLowlight(common);

interface TipTapEditorProps {
  content: string;
  editable: boolean;
  onUpdate?: (markdown: string) => void;
}

export function TipTapEditor({ content, editable, onUpdate }: TipTapEditorProps) {
  const editor = useEditor({
    extensions: [
      StarterKit.configure({
        codeBlock: false, // replaced by CodeBlockLowlight
      }),
      Markdown,
      TaskList,
      TaskItem.configure({ nested: true }),
      Placeholder.configure({ placeholder: 'Empty document' }),
      CodeBlockLowlight.configure({ lowlight }),
    ],
    content,
    contentType: 'markdown',
    editable,
    shouldRerenderOnTransaction: false,
    onUpdate: ({ editor }) => {
      if (onUpdate) {
        onUpdate(editor.getMarkdown());
      }
    },
  });

  // Toggle editable state when prop changes
  useEffect(() => {
    if (editor) editor.setEditable(editable);
  }, [editor, editable]);

  if (!editor) return null;

  return (
    <div className="prose prose-sm prose-zinc max-w-none px-6 py-6">
      <EditorContent editor={editor} />
    </div>
  );
}
```

### diff-compute Utility
```typescript
// Source: diff library docs (diffLines API)
import { diffLines, type Change } from 'diff';

export interface DiffHunk {
  type: 'added' | 'removed' | 'unchanged';
  value: string;
  lineCount: number;
}

export function computeDiff(before: string, after: string): DiffHunk[] {
  const changes: Change[] = diffLines(before, after);
  return changes.map((change) => ({
    type: change.added ? 'added' : change.removed ? 'removed' : 'unchanged',
    value: change.value,
    lineCount: change.count ?? 0,
  }));
}
```

### POST /api/files/save Endpoint
```typescript
// Source: existing files.ts pattern + contracts/api.md
router.post('/save', async (req: Request, res: Response) => {
  const { filePath, content } = req.body;

  if (!filePath || typeof filePath !== 'string') {
    res.status(400).json({ error: 'filePath is required' });
    return;
  }
  if (content === undefined || content === null) {
    res.status(400).json({ error: 'content is required' });
    return;
  }
  if (!filePath.startsWith('specs/')) {
    res.status(403).json({ error: 'Only files within specs/ directory can be saved' });
    return;
  }

  const resolvedPath = resolve(monorepoRoot, filePath);
  const specsRoot = resolve(monorepoRoot, 'specs');
  if (!resolvedPath.startsWith(specsRoot)) {
    res.status(403).json({ error: 'Path traversal not allowed' });
    return;
  }

  try {
    await writeFile(resolvedPath, content, 'utf-8');
    res.json({ saved: true, path: filePath });
  } catch (err: unknown) {
    const message = err instanceof Error ? err.message : 'Unknown error';
    res.status(500).json({ error: `Failed to write file: ${message}` });
  }
});
```

## Open Questions

1. **TipTap 3 + React 19 full compatibility**
   - What we know: @tiptap/react 3.20.5 is published and installs with React 19. TipTap docs note UI Components work best with React 18 but the core editor works with React 19.
   - What's unclear: Whether any subtle rendering issues exist with React 19's concurrent features.
   - Recommendation: Install and test. React 19 compatibility is likely fine for core editor usage (no TipTap UI Components used in this project). If issues arise, shouldRerenderOnTransaction: true may help.

2. **@tiptap/markdown GFM support (task lists, tables)**
   - What we know: Uses MarkedJS internally. MarkedJS supports GFM via options. The extension accepts markedOptions configuration.
   - What's unclear: Whether task list checkboxes round-trip perfectly through parse->edit->serialize.
   - Recommendation: Configure `Markdown.configure({ markedOptions: { gfm: true } })`. Test with a tasks.md file containing checkboxes early in implementation. If markdown serialization breaks task lists, fall back to tiptap-markdown 0.9.0 which uses markdown-it + markdown-it-task-lists.

3. **Diff overlay UX when multiple files change**
   - What we know: Snapshot captures all open tabs. Multiple files could change after a single agent response.
   - What's unclear: Should diff overlays show on all changed tabs simultaneously, or only the active tab?
   - Recommendation: Show diff on whichever tab the user views. Store diff data per tab in Zustand. Apply decoration when tab becomes active and has pending diff data.
